Overview

A document repository built into WordPress

Admin Doc Repo started as a simple password-protected file upload page and grew into a full document management system with expense tracking, access control, deletion history, and a configurable block editor interface. The plugin is distributed on WordPress.org and installable on any WordPress site without modifying theme files or writing code. The frontend is a React and TypeScript application served via a custom Gutenberg block, communicating with a PHP REST API backend.

A full document management system delivered as a single WordPress block — drop it on any page, configure it from the editor sidebar, and it works.

How it works

Session-based access with server-enforced permissions

Visitors authenticate via a password form rendered by the React frontend. On success, the PHP backend sets a session variable recording the access level — full or restricted. Every subsequent REST API request is gated by a permission callback that reads this server-side session value, making the access level impossible to spoof from the client. WordPress administrators bypass the frontend entirely via a separate dashboard file manager. Permissions for each operation — upload, rename, delete, restore, and permanent deletion — are configurable independently from the block editor and stored as WordPress options, enforced server-side regardless of what the frontend renders.

Features

What's under the hood

📁
File management
Upload, rename, delete, and download files through a sortable, filterable table. Files are stored outside the plugin directory and persist across plugin updates, deactivations, and reinstalls.
🧾
Per-file expense tracking
Each file supports multiple expense rows — category, submitter, date, and amount — for cases where a single document contains multiple receipts. Rows are added, edited, and removed inline with optimistic updates and automatic revert on failure. Optional input to select expenses to include / ignore for reimbursement.
🔒
Two-tier access control
Full and restricted access passwords are configured independently. Each operation is individually permission-gated — restricted users can be allowed or blocked from any combination of upload, rename, delete, restore, and permanent deletion, enforced at the API level.
🗑
Deletion history
Deleted files are moved to a server-side trash directory rather than permanently removed. A trash panel shows original filename, deletion date, and full expense metadata at time of deletion, with a one-click restore. Orphaned trash entries are automatically pruned after 30 days.
🎨
Block editor configuration
Categories, submitters, colours, and permissions are all configurable from the Gutenberg block editor sidebar. Colour changes apply immediately to an inline preview. Preset themes provide one-click coordinated colour schemes with custom override support.
⚙️
WordPress admin integration
A separate dashboard file manager gives WordPress administrators direct access without a frontend password. A Settings page under Settings → Doc Repo manages access passwords independently of the block editor.

Technical decisions

Why these choices

Gutenberg block over shortcode
The plugin began as a PHP shortcode but was migrated to a Gutenberg block to resolve JSX runtime compatibility issues with the WordPress bundler and to enable the block editor sidebar for configuration. The block's render.php passes PHP-side configuration to the React frontend via wp_localize_script, keeping server and client data in sync without a separate API call on load.
Server-side session auth
Authentication uses PHP sessions rather than WordPress nonce-only verification. The access level is stored server-side in $_SESSION['sfm_access'] and checked in every REST permission callback, making it impossible for a client to escalate privileges by modifying request parameters or headers.
Optimistic updates
Expense row edits, additions, and removals update local React state immediately before the API call resolves, making interactions feel instant. On API failure, the previous state is restored from a snapshot taken before the optimistic update, and an inline error message is shown without disrupting other rows.
Filename-keyed metadata
File metadata is stored in a single meta.json file keyed by filename. Rename operations migrate the metadata key atomically alongside the filesystem rename, keeping the two in sync. A migration guard in the read path transparently upgrades the legacy single-object format to the current array-of-rows format without requiring a data migration step.
Stable row keys via crypto.randomUUID
Expense rows are keyed by a crypto.randomUUID()-generated identifier assigned at creation time rather than by array index. This prevents React from reusing component instances across row removals, which would cause local edit state — including the newly-added row's auto-opened edit mode — to bleed between rows. The key is stripped before sending rows to the server.
Separate editor API client
The main API client reads window.SFM.apiBase and window.SFM.nonce, which are only available on the published frontend. A separate editorApi.ts uses WordPress's apiFetch utility for calls made from the block editor context, where window.SFM is not defined. This keeps the two execution contexts cleanly separated.
